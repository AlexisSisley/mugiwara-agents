Architecture cmux — 3 approches

Choisis l'approche qui correspond le mieux à ta vision

A

🏗️ Monolithe modulaire

Un seul binaire qui fait tout : daemon + GUI dans le même process.

cmux (single process)
├── PTY Manager (portable-pty, vte parsing)
├── Layout Engine (workspace → tab → split tree)
├── GPU Renderer (wgpu, glyph atlas, grid)
├── OSC Interceptor (Claude Code detection)
├── Session Store (state persistence on disk)
└── Config / Keybinds

✅ Avantages

  • Simple à développer et débugger
  • Pas d'IPC, tout en mémoire directe
  • Un seul binaire à distribuer

❌ Inconvénients

  • Si le GUI crash, les PTY meurent aussi
  • Pas de persistance totale possible
  • Pas d'accès distant (SSH + attach)

⚠️ Incompatible avec ton choix de persistance totale

B

⭐ Client-Serveur classique (recommandé)

Daemon + Client GUI séparés. Le daemon gère les PTY, le client rend l'UI.

cmux-daemon (background process, always running)
├── PTY Manager (spawn, resize, I/O)
├── Session Registry (workspaces, tabs, panes)
├── Ring Buffer Pool (shared mem per PTY)
├── OSC Interceptor (Claude Code signals)
├── Control Socket (commands in/out)
└── State Persistence (disk save/restore)

cmux-gui (connects to daemon)
├── GPU Renderer (wgpu, reads shared mem)
├── Layout Engine (tree → screen rects)
├── Input Handler (keybinds → control socket)
├── Widget System (tabs, statusbar, panels)
└── Theme Engine (fonts, colors, styles)

cmux (CLI, thin client)
└── Control Socket Client (cmux split, cmux new-tab...)

✅ Avantages

  • Persistance totale — GUI crash ≠ perte de sessions
  • CLI scriptable (cmux split-right, cmux send-keys)
  • Possible d'attacher plusieurs clients (SSH remote)
  • Architecture éprouvée (c'est ce que fait tmux)

❌ Inconvénients

  • IPC shared memory + socket à implémenter
  • Deux binaires à builder et synchroniser
  • Complexité de démarrage (daemon auto-start)

✅ Correspond exactement à tes choix

C

🧩 Micro-services locaux

Plusieurs daemons spécialisés qui communiquent entre eux.

cmux-pty-server (PTY lifecycle only)
cmux-session-server (workspace/tab/split state)
cmux-osc-watcher (Claude Code signal extraction)
cmux-agent-broker (Mugiwara routing, future)
cmux-gui (rendering, connects to all)
cmux (CLI, connects to session-server)

✅ Avantages

  • Chaque service testable et remplaçable isolément
  • Extensibilité maximale (plugin = nouveau service)
  • Crash d'un service n'affecte pas les autres

❌ Inconvénients

  • Over-engineering pour un outil personnel
  • Latence IPC multipliée (N services × M connexions)
  • Orchestration complexe (qui démarre quoi ?)
  • Debugging distribué, même en local

⚡ Puissant mais probablement excessif au début
