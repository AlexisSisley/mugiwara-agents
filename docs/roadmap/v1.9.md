# v1.9 — Routage Intelligent, Renommages OP & Nouveaux Agents Metier

**Date** : Mars 2026
**Capitaine** : Luffy (Program Manager)

---

## 1. Resume Executif

La v1.8 a complete l'ecosysteme avec 46 agents uniques couvrant l'infrastructure (Docker, IIS, Firebase, reseau). La v1.9 franchit un cap strategique en trois axes : **(1) Renommage** des 2 agents sans nom One Piece et des 8 pipelines en noms de navires, **(2) 10 nouveaux agents metier** couvrant les domaines IT manquants (Cloud AWS/Azure/GCP, Event-Driven, Accessibilite, AI/ML Ops, Agile, BI, DBA, Chaos Engineering), et **(3) Routage intelligent inter-agents** — chaque agent possede une table de redirection vers ses agents complementaires, formant une toile complete ou aucune demande ne tombe dans le vide. Volume projete : **56 agents uniques + aliases**.

## 4. Axe 3 — Routage Intelligent Inter-Agents

### 4.1 Concept

Chaque agent possede dans son SKILL.md une **table de routage** vers les agents complementaires. Quand un agent detecte qu'une partie de la demande sort de son perimetre, il suggere (ou invoque via `Skill`) l'agent le plus adapte. Cela forme une **toile complete** ou aucun domaine IT n'est orphelin.

### 4.2 Matrice de routage (extrait)

| Agent source | Redirige vers | Condition de routage |
|-------------|--------------|---------------------|
| `/zorro` | `/sanji` | Specs validees, besoin d'architecture |
| `/sanji` | `/sanji-<stack>` | Stack choisie, besoin d'implementation |
| `/sanji` | `/crocodile`, `/kizaru`, `/aokiji` | Besoin cloud specifique detecte |
| `/chopper` | `/enel` | Bug diagnostique, besoin de monitoring |
| `/franky` | `/fujitora` | Problemes d'accessibilite detectes |
| `/law` | `/hawkins` | Donnees prates, besoin de visualisation |
| `/law` | `/magellan` | Performance DB, besoin d'optimisation |
| `/usopp` | `/crocodile`, `/kizaru`, `/aokiji` | Deploiement cloud specifique |
| `/usopp` | `/iceburg` | Besoin de containerisation |
| `/doflamingo` | `/law` | Event sourcing, besoin data pipeline |
| `/katakuri` | `/law` | ML pipeline, besoin data engineering |
| `/katakuri` | `/usopp` | Model serving, besoin infra |
| `/caesar` | `/enel` | Post-chaos, besoin monitoring |
| `/caesar` | `/ace` | Resilience, besoin performance |
| `/big-mom` | `/vivi` | Sprint planning, besoin product |
| `/big-mom` | `/zorro` | Sprint backlog, besoin specs |
| `/magellan` | `/jinbe` | Securite DB, besoin audit |
| `/sabo` | `/sanji-ts` | Firebase + frontend, besoin TypeScript |
| `/coby` | `/jinbe` | Firewall rules, besoin securite |
| `/one_piece` | **Tous** | Routeur central universel |

### 4.3 Implementation technique

- Ajout de `Skill` dans les `allowed-tools` de chaque agent
- Ajout d'une section `## Routage Inter-Agents` dans chaque SKILL.md
- Le routage est **suggestif** (l'agent propose) sauf dans les pipelines ou il est **automatique**
- `/one_piece` reste le routeur central qui connait la toile complete
