# v1.1 — Fix disable-model-invocation

**Date** : 2026-03-02

---

## Probleme

Quand un utilisateur lancait `/one_piece` ou un pipeline (`/mugiwara`, `/incident`, etc.) **depuis un autre projet** (ex: MERCH_BO_dev), les sous-agents n'etaient pas invocables via l'outil `Skill`. Le pipeline se rabattait alors sur lui-meme pour faire le travail au lieu de deleguer aux agents specialises.

**Message d'erreur typique :**
> Les skills individuelles (mugiwara, zorro, sanji, nami, luffy) n'etaient pas disponibles via l'outil Skill dans cet environnement.

## Cause racine

Tous les SKILL.md contenaient dans leur front matter YAML :

```yaml
disable-model-invocation: true
```

Ce parametre **empeche le modele d'invoquer une skill programmatiquement** via l'outil `Skill`. Seul l'utilisateur peut l'appeler en tapant `/commande` manuellement.

Or quand un pipeline (ex: `/mugiwara`) execute `Skill(skill: "zorro", args: "...")`, c'est **le modele** qui fait l'appel — et `disable-model-invocation: true` le bloque.

### Schema du probleme

```
/one_piece "mon probleme"
  → one_piece s'execute (context: fork)
    → Skill(skill: "zorro")
      → BLOQUE car disable-model-invocation: true
    → l'agent se rabat sur lui-meme (mauvais comportement)
```

## Solution

Passage de `disable-model-invocation: true` a `disable-model-invocation: false` dans **tous les SKILL.md** (front matter YAML uniquement).

### Schema apres fix

```
/one_piece "mon probleme"
  → one_piece s'execute (context: fork)
    → Skill(skill: "zorro") → OK
    → Skill(skill: "mugiwara") → OK
      → Skill(skill: "sanji") → OK
      → Skill(skill: "nami") → OK
      → Skill(skill: "luffy") → OK
```
