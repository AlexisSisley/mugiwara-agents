# v1.7 — Nouveaux Agents & Pipelines

**Date** : Mars 2026
**Capitaine** : Luffy (Program Manager)

---

## 1. Resume Executif

La v1.6 a livre un dashboard web complet (Svelte + Express + TypeScript, 116 tests, couverture >97%) qui rend l'ecosysteme Mugiwara observable et demonstrable. Cependant, l'ecosysteme reste **reactif** : on voit les problemes apres qu'ils se produisent. La v1.7 passe a l'**etape proactive** en ajoutant un agent Monitoring/Alerting (Prometheus, Grafana, PagerDuty), un agent Feature Flags (LaunchDarkly, Unleash, env-based), un pipeline Release automatise (changelog + morgans + tag + deploy), et un audit de sante global de l'ecosysteme. En parallele, trois items reportes de la v1.6 — CHANGELOG automatique, Conventional Commits + commitlint, et schema JSON standardise des logs — sont integres pour consolider la maturite du projet. L'approche recommandee est un MVP en 2 semaines (items reportes + agent monitoring), suivi de deux phases d'enrichissement (feature flags, pipeline release, audit).

---

## 2. Synthese d'Alignement

**3 points cles d'alignement :**

1. **Les items reportes de la v1.6 sont des fondations** : le schema JSON des logs, Conventional Commits et le CHANGELOG automatique sont des pre-requis pour le pipeline Release et l'audit ecosysteme. Les livrer en premier debloque toute la suite.
2. **Le monitoring comble la lacune majeure de l'ecosysteme** : avec 40 agents et un dashboard, l'absence d'alerting est le risque numero un. Un incident non detecte peut passer inapercu des heures.
3. **Le pipeline Release automatise le cycle de livraison** : aujourd'hui chaque release est manuelle (tag, changelog, communication). L'automatiser reduit le cycle time et elimine les oublis.

## 10. Actions Immediates

### Action 1 : Installer commitlint et configurer le hook pre-commit

**Responsable** : `/usopp` (hook) + `/brook` (documentation)
**Delai** : Jour 1

Installer `@commitlint/cli` et `@commitlint/config-conventional` a la racine du projet. Configurer un hook pre-commit (via Husky ou simple git hook) qui valide le format des commits. Mettre a jour `CONTRIBUTING.md` avec les conventions (`feat:`, `fix:`, `docs:`, `chore:`, `test:`, `ci:`). C'est le pre-requis bloquant pour le CHANGELOG automatique.

### Action 2 : Definir et implementer le schema JSON des logs JSONL

**Responsable** : `/nami` (specification) + `/usopp` (integration hooks)
**Delai** : Jour 1-2

Creer un fichier `schemas/agent-event.schema.json` definissant la structure des events dans les logs JSONL :
```json
{
  "timestamp": "ISO-8601",
  "type": "agent_invocation | session_start | session_end | pipeline_complete",
  "agent": "string",
  "pipeline": "string | null",
  "duration_ms": "number",
  "status": "success | error",
  "version": "string",
  "metadata": {}
}
```
Integrer un validateur dans les hooks v1.3 existants. Ce schema est le contrat entre les hooks, le dashboard, et le futur pipeline Release.

### Action 3 : Scaffolder l'agent Monitoring/Alerting

**Responsable** : `/vegapunk`
**Delai** : Jour 2-3

Creer le dossier `skills/monitoring/` avec :
- `SKILL.md` : agent couvrant Prometheus (config, rules, scraping), Grafana (dashboards, alerting), bonnes pratiques
- `mugiwara.yaml` : manifest avec version 1.7.0, categorie `infrastructure`, checksum SHA256
- Mise a jour de `registry.yaml` pour inclure le nouvel agent

Lancer en parallele des Actions 1 et 2 pour optimiser le calendrier.
