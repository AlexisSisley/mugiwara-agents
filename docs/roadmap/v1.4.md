# v1.4 — Tests fonctionnels & CI/CD

**Date** : Mars 2026

---

## Apercu

La v1.4 fait passer le projet du stade "ca fonctionne" au stade "c'est prouve". Les 342 smoke tests structurels de la v1.2 sont completes par des **tests fonctionnels** qui executent reellement les agents. Un pipeline **GitHub Actions** automatise l'ensemble a chaque push/PR.

---

## Objectifs

| # | Objectif | KPI | Definition of Done |
|---|----------|-----|--------------------|
| 1 | Tests fonctionnels agents | 40 agents testes, 0 echec | Chaque agent execute avec un prompt minimal, output valide (non-vide, format attendu, pas d'erreur) |
| 2 | GitHub Actions CI | Pipeline vert sur main | Smoke tests + tests fonctionnels executent a chaque push/PR, badge status dans README |
| 3 | Validation hooks automatisee | 6 hooks testes | Tests automatises pour les 6 hooks v1.3 (log, validate, notify, session, smoke, complete) |
| 4 | Politique SemVer | Document publie | Fichier VERSIONING.md definissant MAJOR/MINOR/PATCH avec exemples |

## Taches detaillees

### 1. Framework de tests fonctionnels

**Agent** : `/nami` (QA Lead)

**Description** : Creer un framework de test qui :
- Execute chaque agent avec un prompt de test minimal (ex: `/zorro "Genere un test unitaire pour une fonction add(a, b)"`)
- Valide la structure de l'output (non-vide, longueur minimale, pas de message d'erreur)
- Produit un rapport de resultats au format JUnit XML (compatible CI)
- Tourne en < 10 minutes pour les 40 agents

**Fichiers a creer** :

| Fichier | Role |
|---------|------|
| `tests/functional/run-functional-tests.sh` | Runner principal des tests fonctionnels |
| `tests/functional/test-prompts.json` | Prompts de test par agent (1 prompt minimal par agent) |
| `tests/functional/validate-output.sh` | Validateur generique d'output agent |
| `tests/functional/results/` | Dossier de resultats (gitignored) |

### 2. GitHub Actions CI

**Agent** : `/usopp` (DevOps / CI-CD)

**Description** : Pipeline CI avec :
- Trigger sur push main + PR
- Job 1 : Smoke tests existants (`tests/smoke/`)
- Job 2 : Tests fonctionnels (`tests/functional/`)
- Job 3 : Lint des SKILL.md (YAML valide, champs obligatoires)
- Badge status dans le README.md
- Artefacts : rapport de tests

**Fichiers a creer** :

| Fichier | Role |
|---------|------|
| `.github/workflows/ci.yml` | Pipeline CI principal |
| `.github/workflows/lint-skills.yml` | Lint des fichiers SKILL.md (optionnel, peut etre dans ci.yml) |

### 3. Tests hooks v1.3

**Agent** : `/nami` (QA Lead)

**Description** : Tests automatises pour chaque hook :

| Hook | Test |
|------|------|
| `log-agent-output.sh` | Simuler un event PostToolUse → verifier ecriture dans `logs/agents.jsonl` |
| `validate-agent-output.sh` | Simuler un output vide → verifier retour d'erreur sync |
| `log-session.sh` | Simuler SessionStart/End → verifier `logs/sessions.jsonl` |
| `notify-slack.sh` | Simuler avec mock webhook → verifier appel curl |
| `notify-complete.sh` | Simuler Stop event → verifier log + notification |
| `run-smoke-tests.sh` | Modifier un fichier skills/ → verifier execution tests |

**Fichiers a creer** :

| Fichier | Role |
|---------|------|
| `tests/hooks/test-hooks.sh` | Suite de tests pour les 6 hooks |
| `tests/hooks/mock-event.json` | Fixtures d'events Claude Code simules |

## Risques identifies

| # | Risque | Impact | Probabilite | Mitigation |
|---|--------|--------|-------------|------------|
| 1 | Tests fonctionnels trop lents (execution reelle des agents) | CI > 30 min | Moyenne | Paralleliser les tests, limiter a 1 prompt par agent |
| 2 | Agents necessitant un contexte projet pour fonctionner | Tests echouent sans repo cible | Haute | Creer un repo fixture minimal dans `tests/fixtures/` |
| 3 | GitHub Actions — cout des runners pour execution LLM | Budget CI | Moyenne | Utiliser self-hosted runner ou limiter aux PR |
| 4 | Hooks dependant de `jq` non present sur tous les runners | CI echoue | Faible | Installer `jq` dans le workflow CI |
