# v1.6 — Dashboard & Observabilite

**Date** : Mars 2026
**Capitaine** : Luffy (Program Manager)

---

## 1. Resume Executif

Le projet Mugiwara Agents a atteint une maturite remarquable en 5 sprints : **40 agents specialises**, 8 pipelines orchestres, un CLI plugin manager complet, un systeme de manifests avec integrite SHA256, un pipeline CI/CD a 4 jobs, et des hooks d'automatisation post-agent. L'ecosysteme produit deja des donnees riches (logs JSONL, sessions, invocations) mais **personne ne les voit**. La v1.6 resout ce probleme en construisant un **Dashboard Web de visualisation** et un **systeme d'observabilite** qui transforment les logs muets en intelligence operationnelle actionnable. L'approche recommandee est un MVP leger (SPA statique + API minimale lisant les JSONL) livrable en 2 semaines, suivi d'enrichissements progressifs (temps reel, metriques avancees, CHANGELOG automatique).

---

## 2. Synthese d'Alignement

**3 points cles d'alignement :**

1. **Les donnees existent deja** : les hooks v1.3 produisent des logs JSONL structures. Le cout d'entree est faible car on consomme de l'existant.
2. **La visibilite est le prochain multiplicateur de valeur** : un dashboard rend l'ecosysteme demonstrable aux parties prenantes et accelere le debug/monitoring.
3. **L'architecture plugin (v1.5) permet une livraison modulaire** : le dashboard peut etre un agent/plugin autonome, coherent avec le systeme existant.

### Matrice d'alignement inter-fonctionnel

| Dimension | Vue Business (Zorro) | Vue Technique (Sanji) | Vue Qualite (Nami) | Consensus |
|-----------|---------------------|----------------------|--------------------|-----------|
| **Perimetre** | Dashboard visible, metriques d'adoption, ROI demontrable | SPA legere, API REST, lecture JSONL, pas de base de donnees | Couverture de tests pour le dashboard, pas de regression sur l'existant | MVP avec 3 vues cles (agents, pipelines, sessions) + tests |
| **Priorites** | Vues business-friendly en premier (KPIs, succes/echec) | Architecture propre, extensible, decouplage frontend/backend | Tests E2E du dashboard, validation des donnees affichees | Dashboard fonctionnel > dashboard joli |
| **Delais** | Livrable sous 2 semaines pour la demo stakeholder | 1 semaine API + 1 semaine frontend est realiste | +2 jours pour les tests du dashboard | MVP en 2 semaines, enrichissements en semaine 3 |
| **Qualite** | Donnees fiables et a jour | Code type-safe (TypeScript), CI integree | 80% couverture tests, 0 regression CI existante | TypeScript + tests obligatoires + CI etendue |
| **Ressources** | 1 developpeur frontend suffit | sanji-ts (frontend), sanji-python ou sanji-ts (API), usopp (deploy) | nami (tests), franky (code review) | Equipe de 3-4 agents specialises |

**Convergences** : les 3 vues s'accordent sur un MVP leger sans base de donnees, livrable rapidement. **Divergence principale** : Zorro veut des metriques business des le MVP, Sanji veut poser une architecture extensible d'abord, Nami veut des tests avant tout deploiement.

## 9. Journal de Decisions

| Decision | Contexte | Alternatives Considerees | Justification |
|----------|---------|-------------------------|---------------|
| **Svelte + Vite pour le frontend** | Besoin d'un framework leger pour une SPA de monitoring | React/Next.js (lourd, overkill), Vue 3 (viable), vanilla JS (pas maintenable) | Svelte offre le meilleur ratio performance/DX pour une app de taille moyenne. Bundle minimal. |
| **Pas de base de donnees en MVP** | Les logs JSONL existent deja, pas besoin de persistence supplementaire | SQLite (simple), PostgreSQL (overkill), fichiers plats (choisi) | Eliminer toute complexite non essentielle. Les JSONL sont deja structures et suffisants pour le volume actuel. |
| **Polling 30s au lieu de WebSocket** | Besoin de rafraichir les donnees sans reload | WebSocket (complexe), SSE (moyen), polling (simple) | Le volume de donnees ne justifie pas la complexite temps reel. Polling est testable et previsible. |
| **GitHub Pages pour le hosting** | Besoin d'un URL accessible pour la demo | Vercel (gratuit aussi), Netlify (gratuit aussi), self-hosted (cout) | Zero configuration supplementaire, deja dans l'ecosysteme GitHub du projet. |
| **Conventional Commits adoptes en v1.6** | Pre-requis pour le CHANGELOG automatique en Phase 2 | Aucune convention (statu quo), format libre avec tags | Standard de l'industrie, tooling mature (commitlint, standard-version), retrocompatible. |
| **CHANGELOG automatique via Brook** | Besoin de communication de release professionnelle | Genere manuellement par morgans, changelog-from-git-log | Brook est deja le Technical Writer de l'equipe. Automatiser via conventional-changelog-cli. |
| **Job CI separe pour le dashboard** | Ne pas impacter les 4 jobs CI existants | Integrer dans un job existant, pipeline separe | Isolation des risques. Si le dashboard casse, les tests agents continuent de passer. |

## 10. Actions Immediates

### Action 1 : Adopter Conventional Commits des maintenant

**Responsable** : `/brook` (redaction guide) + `/usopp` (hook commitlint)
**Delai** : Jour 1

Creer un fichier `CONTRIBUTING.md` avec les conventions de commit :
- `feat:` pour nouvelles fonctionnalites
- `fix:` pour corrections de bugs
- `docs:` pour documentation
- `chore:` pour maintenance
- `test:` pour ajouts de tests

Installer un hook pre-commit `commitlint` pour valider le format. C'est le pre-requis pour le CHANGELOG automatique en Phase 2.

### Action 2 : Definir le schema JSON des logs

**Responsable** : `/nami` (specification) + `/sanji-ts` (implementation)
**Delai** : Jour 1-2

Standardiser le format des events dans `logs/agents.jsonl` et `logs/sessions.jsonl` :
```json
{
  "timestamp": "ISO-8601",
  "type": "agent_invocation | session_start | session_end | pipeline_complete",
  "agent": "nom-agent",
  "pipeline": "nom-pipeline | null",
  "duration_ms": 1234,
  "status": "success | error",
  "metadata": {}
}
```

Ajouter un validateur JSON Schema dans les hooks v1.3 existants. Ce schema est le contrat entre les hooks et l'API du dashboard.
