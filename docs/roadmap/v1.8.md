# v1.8 — Agents Infrastructure, Docker, IIS & Firebase

**Date** : Avril 2026
**Capitaine** : Luffy (Program Manager)

---

## 1. Resume Executif

La v1.7 a consolide l'ecosysteme Mugiwara avec 42 agents, un pipeline Release automatise, le monitoring Prometheus/Grafana et les feature flags. Cependant, l'ecosysteme ne couvre pas encore les besoins fondamentaux d'infrastructure : gestion reseau, containerisation Docker, deploiement IIS et services cloud Firebase. La v1.8 comble cette lacune critique en ajoutant quatre agents specialises — **Infrastructure Reseau** (VLAN, firewall, DNS, load balancing, VPN), **Docker** (Dockerfile, docker-compose, orchestration Swarm/K8s), **IIS** (configuration, deploiement, pools applicatifs, certificats SSL) et **Firebase** (Authentication, Firestore, Realtime Database, Cloud Functions, Hosting, Storage, Security Rules, FCM). Ces agents permettent a l'equipage Mugiwara de couvrir le cycle complet du deploiement applicatif, de l'ecriture du code jusqu'a la mise en production sur l'infrastructure cible — que ce soit on-premise ou cloud. L'approche recommandee est un MVP en 2 semaines (agent Docker + agent IIS + agent Firebase de base), suivi d'un enrichissement en Phase 2 (agent Infrastructure Reseau + orchestration avancee + Firebase avance) et d'une phase d'integration/audit en Phase 3.

---

## 2. Synthese d'Alignement

**3 points cles d'alignement :**

1. **L'infra est le dernier maillon manquant** : les agents existants couvrent le code, les tests, le CI/CD, le monitoring et les feature flags. L'absence d'agents infra/Docker/IIS/Firebase force les equipes a quitter l'ecosysteme Mugiwara pour ces taches, rompant le flux de travail.
2. **Docker est devenu un standard universel** : tout projet moderne utilise des conteneurs. Un agent Docker de qualite a un impact immediat sur l'ensemble des equipes.
3. **IIS reste incontournable dans les environnements Windows/.NET** : de nombreuses applications legacy et nouvelles tournent sur IIS. Cet agent repond a un besoin reel non couvert par les outils generiques.
4. **Firebase est la plateforme BaaS dominante** : Authentication, Firestore, Cloud Functions, Hosting — Firebase couvre le backend-as-a-service pour les projets web et mobile. Un agent Firebase specialise accelere drastiquement le setup, la configuration des Security Rules, et l'integration des services Firebase dans n'importe quel stack.

## 9. Journal de Decisions

| Decision | Contexte | Alternatives Considerees | Justification |
|----------|---------|-------------------------|---------------|
| **4 agents distincts (docker, iis, infra-reseau, firebase) plutot qu'un mega-agent infra/cloud** | Besoin de couvrir Docker, IIS, l'infra reseau et Firebase | Un seul agent "infrastructure" couvrant tout, 2 agents (docker+iis et reseau+firebase), 3 agents sans Firebase | Un agent par domaine est plus coherent avec l'architecture existante (un agent = une expertise). Firebase est un domaine suffisamment riche et distinct pour meriter son propre agent. Plus facile a maintenir, tester et faire evoluer independamment. |
| **Docker, IIS et Firebase en parallele en Phase 1** | Les trois agents sont independants et a forte demande | Sequentiel (Docker puis IIS puis Firebase), les 4 en parallele | Docker, IIS et Firebase n'ont pas de dependances mutuelles. Les developper en parallele optimise le calendrier. L'infra reseau est plus complexe et beneficie d'une Phase 2 dediee. |
| **sanji-dotnet comme co-responsable de l'agent IIS** | IIS est specifique a l'ecosysteme Windows/.NET | vegapunk seul, creer un nouvel agent dedie | sanji-dotnet a l'expertise .NET necessaire pour les web.config, PowerShell et les patterns de deploiement IIS. Repartir la charge reduit le risque R5. |
| **sanji-ts comme co-responsable de l'agent Firebase** | Firebase est un ecosysteme JavaScript/TypeScript natif | vegapunk seul, creer un nouvel agent dedie | sanji-ts a l'expertise TypeScript/JavaScript necessaire pour les Cloud Functions, le SDK Firebase v9 modular, et les patterns frontend. Repartir la charge reduit le risque R11 (surcharge vegapunk). |
| **Integration Docker + IIS en Phase 3** | Les conteneurs Windows IIS sont un cas d'usage avance | En Phase 1 avec Docker, pas d'integration | L'integration necessite que les deux agents soient matures. La Phase 3 est le bon moment pour combiner les expertises. |
| **Firewall + DNS + LB comme piliers de l'agent reseau** | Le domaine "infrastructure reseau" est tres large | Tout couvrir (VLAN, VPN, SD-WAN, MPLS, BGP...), uniquement firewall | Les 3 piliers (firewall, DNS, load balancing) couvrent 80% des besoins quotidiens des equipes. VPN et VLAN sont ajoutes en extension Phase 3. |
| **Templates avec avertissements de securite** | Les configurations reseau/firewall peuvent etre dangereuses si mal appliquees | Pas d'avertissement (confiance utilisateur), bloquer les commandes destructrices | Compromis : eduquer l'utilisateur avec des annotations claires sans bloquer les cas d'usage legitimes. Mode dry-run par defaut pour les operations critiques. |
