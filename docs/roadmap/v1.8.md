# v1.8 — Agents Infrastructure, Docker & IIS

**Date** : Avril 2026
**Capitaine** : Luffy (Program Manager)

---

## 1. Resume Executif

La v1.7 a consolide l'ecosysteme Mugiwara avec 42 agents, un pipeline Release automatise, le monitoring Prometheus/Grafana et les feature flags. Cependant, l'ecosysteme ne couvre pas encore les besoins fondamentaux d'infrastructure : gestion reseau, containerisation Docker et deploiement IIS. La v1.8 comble cette lacune critique en ajoutant trois agents specialises — **Infrastructure Reseau** (VLAN, firewall, DNS, load balancing, VPN), **Docker** (Dockerfile, docker-compose, orchestration Swarm/K8s) et **IIS** (configuration, deploiement, pools applicatifs, certificats SSL). Ces agents permettent a l'equipage Mugiwara de couvrir le cycle complet du deploiement applicatif, de l'ecriture du code jusqu'a la mise en production sur l'infrastructure cible. L'approche recommandee est un MVP en 2 semaines (agent Docker + agent IIS de base), suivi d'un enrichissement en Phase 2 (agent Infrastructure Reseau + orchestration avancee) et d'une phase d'integration/audit en Phase 3.

---

## 2. Synthese d'Alignement

**3 points cles d'alignement :**

1. **L'infra est le dernier maillon manquant** : les agents existants couvrent le code, les tests, le CI/CD, le monitoring et les feature flags. L'absence d'agents infra/Docker/IIS force les equipes a quitter l'ecosysteme Mugiwara pour ces taches, rompant le flux de travail.
2. **Docker est devenu un standard universel** : tout projet moderne utilise des conteneurs. Un agent Docker de qualite a un impact immediat sur l'ensemble des equipes.
3. **IIS reste incontournable dans les environnements Windows/.NET** : de nombreuses applications legacy et nouvelles tournent sur IIS. Cet agent repond a un besoin reel non couvert par les outils generiques.

### Matrice d'alignement inter-fonctionnel

| Dimension | Vue Business (Zorro) | Vue Technique (Sanji) | Vue Qualite (Nami) | Consensus |
|-----------|---------------------|----------------------|--------------------|-----------|
| **Perimetre** | 3 agents a forte valeur operationnelle couvrant l'infra de bout en bout | Architecture modulaire : un agent par domaine (reseau, Docker, IIS), SKILL.md riches avec exemples concrets | Tests pour chaque agent, templates valides, pas de regression sur les 42 agents existants | 3 agents (infra-reseau, docker, iis) + tests + integration registry |
| **Priorites** | Docker en premier (adoption la plus large), puis IIS (clients Windows), puis reseau (specialise) | Docker d'abord (moins de dependances), IIS ensuite (Windows-specific), reseau en dernier (plus complexe) | Validation des templates generes (Dockerfile, docker-compose, web.config) avant tout enrichissement | Docker -> IIS -> Infrastructure Reseau |
| **Delais** | Livrable sous 4 semaines pour maintenir le rythme mensuel | 1 semaine par agent est realiste, +1 semaine pour integration | +2 jours par agent pour les tests et la validation des outputs | MVP en 2 semaines, V1 en semaine 3, V2 en semaine 4 |
| **Qualite** | Agents fonctionnels, documentes, avec exemples prets a l'emploi | Code coherent avec l'architecture existante, manifests mugiwara.yaml standardises | 80% couverture, templates valides syntaxiquement, audit vegapunk | Tests obligatoires + CI etendue + audit final |
| **Ressources** | vegapunk (agents), equipe existante pour support | vegapunk (agents infra), sanji-dotnet (expertise IIS/.NET), usopp (CI) | nami (QA), franky (review), vegapunk (audit) | Equipe de 5 agents specialises |

**Convergences** : les 3 vues s'accordent sur la priorite de Docker et la necessite de templates valides. **Divergence principale** : Zorro veut livrer les 3 agents rapidement pour maximiser l'impact business, Sanji veut des SKILL.md tres detailles avec des exemples d'orchestration avancee (K8s, Swarm), Nami veut valider chaque template genere avant de passer a l'agent suivant.

## 3. Arbitrage des Conflits

| Conflit | Position Zorro | Position Sanji | Position Nami | Decision d'Arbitrage | Justification |
|---------|---------------|----------------|---------------|---------------------|---------------|
| **Scope agent Docker** | Dockerfile + docker-compose + K8s + Swarm des le MVP | Docker multi-stage + compose v2 + orchestration modulaire | Validation syntaxique de chaque Dockerfile et compose genere | **Dockerfile + docker-compose en MVP. Orchestration (Swarm, K8s) en Phase 2.** | Lean MVP : Dockerfile et compose couvrent 90% des cas d'usage. L'orchestration est une extension naturelle. |
| **Scope agent IIS** | Configuration complete + deploiement + SSL + pools + URL Rewrite | Architecture modulaire : web.config, applicationHost.config, PowerShell scripts | Tests de validite des configurations XML generees | **web.config + pools + SSL en MVP. URL Rewrite + ARR + deploiement avance en Phase 2.** | Le coeur d'IIS (sites, pools, SSL) couvre les besoins immediats. Les modules avances sont des extensions. |
| **Scope agent Infrastructure Reseau** | Veut tout (VLAN, firewall, DNS, LB, VPN, SD-WAN) | Prefere se concentrer sur les patterns les plus courants (firewall rules, DNS, reverse proxy) | Veut des exemples valides et testables pour chaque pattern | **Firewall + DNS + Load Balancing en MVP. VPN + VLAN + SD-WAN en extension.** | Les 3 piliers reseau les plus utilises en premier. Les cas avances en option. |
| **Ordre de livraison** | Les 3 agents en parallele pour aller vite | Docker d'abord (moins de risque), sequentiel | Un agent a la fois, teste avant de passer au suivant | **Docker et IIS en parallele (Phase 1), Infrastructure Reseau en Phase 2.** | Docker et IIS sont independants, les developper en parallele optimise le calendrier sans sacrifier la qualite. |
| **Niveau de detail des SKILL.md** | Exemples prets a copier-coller, orientes productivite | Documentation exhaustive avec explications des choix techniques | Exemples corrects et valides, pas de code qui ne compile pas | **SKILL.md riches avec exemples concrets ET annotations explicatives. Chaque exemple doit etre syntaxiquement valide.** | Compromis : productivite (copier-coller) + pedagogie (annotations) + qualite (syntaxe valide). |

## 9. Journal de Decisions

| Decision | Contexte | Alternatives Considerees | Justification |
|----------|---------|-------------------------|---------------|
| **3 agents distincts (docker, iis, infra-reseau) plutot qu'un mega-agent infra** | Besoin de couvrir Docker, IIS et l'infra reseau | Un seul agent "infrastructure" couvrant tout, 2 agents (docker+iis et reseau) | Un agent par domaine est plus coherent avec l'architecture existante (un agent = une expertise). Plus facile a maintenir, tester et faire evoluer independamment. |
| **Docker et IIS en parallele en Phase 1** | Les deux agents sont independants et a forte demande | Sequentiel (Docker puis IIS), les 3 en parallele | Docker et IIS n'ont pas de dependances mutuelles. Les developper en parallele optimise le calendrier. L'infra reseau est plus complexe et beneficie d'une Phase 2 dediee. |
| **sanji-dotnet comme co-responsable de l'agent IIS** | IIS est specifique a l'ecosysteme Windows/.NET | vegapunk seul, creer un nouvel agent dedie | sanji-dotnet a l'expertise .NET necessaire pour les web.config, PowerShell et les patterns de deploiement IIS. Repartir la charge reduit le risque R5. |
| **Integration Docker + IIS en Phase 3** | Les conteneurs Windows IIS sont un cas d'usage avance | En Phase 1 avec Docker, pas d'integration | L'integration necessite que les deux agents soient matures. La Phase 3 est le bon moment pour combiner les expertises. |
| **Firewall + DNS + LB comme piliers de l'agent reseau** | Le domaine "infrastructure reseau" est tres large | Tout couvrir (VLAN, VPN, SD-WAN, MPLS, BGP...), uniquement firewall | Les 3 piliers (firewall, DNS, load balancing) couvrent 80% des besoins quotidiens des equipes. VPN et VLAN sont ajoutes en extension Phase 3. |
| **Templates avec avertissements de securite** | Les configurations reseau/firewall peuvent etre dangereuses si mal appliquees | Pas d'avertissement (confiance utilisateur), bloquer les commandes destructrices | Compromis : eduquer l'utilisateur avec des annotations claires sans bloquer les cas d'usage legitimes. Mode dry-run par defaut pour les operations critiques. |
